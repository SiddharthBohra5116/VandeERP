CRITICAL

Password reset accepts empty or blank passwords — admin can lock out any user

adminController.js → resetPassword
AuthInput validation
⌄

The resetPassword handler assigns req.body.password directly with zero validation. An admin can submit a blank form and it will hash an empty string, locking the user out. There is also no null-check on the user lookup — the function crashes with a TypeError if the user was deleted between the request being made and the admin approving it.

// CURRENT — no guards at all exports.resetPassword = async (req, res) => { const u = await User.findById(req.params.id); // crashes if user deleted u.password = req.body.password; // could be "", undefined, or "a" u.resetRequested = false; await u.save(); }; // FIX exports.resetPassword = async (req, res) => { const { password } = req.body; if (!password || password.trim().length < 8) return res.redirect('/admin/dashboard?error=Password+must+be+at+least+8+characters'); const u = await User.findById(req.params.id); if (!u) return res.redirect('/admin/dashboard?error=User+not+found'); u.password = password.trim(); u.resetRequested = false; await u.save(); };
  • Add minimum 8-character check with .trim() on the password input
  • Add a null-check on the user lookup before accessing any property
  • Log the reset action with a timestamp and the admin's user ID for the audit trail
CRITICAL

Unescaped MongoDB $regex on user input — ReDoS risk and data probing

adminController.js · counsellorController.js — 6 locations
ReDoSData exposure
⌄

Every search field passes raw user input directly into { $regex: search, $options: 'i' }. A crafted input such as (a+)+b causes catastrophic backtracking and can exhaust CPU (ReDoS). Users can also probe exact data by treating the search string as a regex pattern.

filter.name = { $regex: search, $options: 'i' }; // raw, unescaped user input // utils/sanitize.js function escapeRegex(str) { return str.replace(/[.*+?^${}()|[\]\\]/g, '\\$&'); } module.exports = { escapeRegex }; // Usage in controllers: filter.name = { $regex: escapeRegex(search), $options: 'i' };
  • Create utils/sanitize.js with an escapeRegex() helper
  • Import and apply it in all 6 places that currently use raw $regex
  • Add a max-length guard (50 characters) on all search query parameters
CRITICAL

No CSRF protection on any form — every POST route is cross-site forgeable

All route files — no csurf or equivalent middleware installed
CSRFAll roles
⌄

Zero CSRF token middleware exists anywhere in the codebase. Every POST route — password reset, payment recording, user creation, lead conversion, status changes — can be triggered by a cross-site request if the target user is logged in. An attacker embeds a hidden form on any page they control and it fires silently when an admin or teacher visits it.

// app.js — add after cookie-parser const csurf = require('csurf'); app.use(csurf({ cookie: true })); app.use((req, res, next) => { res.locals.csrfToken = req.csrfToken(); next(); }); // Every EJS form — add inside <form>: <input type="hidden" name="_csrf" value="<%= csrfToken %>">
  • Install csurf middleware and register it globally after cookie-parser
  • Expose csrfToken via res.locals so it is available in every EJS template
  • Add the hidden CSRF input to every <form> in every EJS template
  • Send an X-CSRF-Token header for any AJAX calls that mutate state

Fee payment double-writes paidAmount — pre-save hook + manual increment = corrupted ledger

adminController.js → postAddPayment · Fee model pre-save hook
Data integrityFee model
⌄

postAddPayment manually increments fee.paidAmount += Number(amount) and then calls fee.save(). The Fee model's pre-save hook runs allocatePayments(), which recomputes paidAmount from the payment array sum — overwriting the manual value. The sync to User.fees_paid runs before the hook corrects things, meaning it captures the inflated pre-hook value and writes incorrect data to the User document.

fee.payments.push({ amount: Number(amount), ... }); fee.paidAmount += Number(amount); // ← redundant and incorrect await fee.save(); // hook recalculates from scratch (overwrites above) await User.findByIdAndUpdate(..., { fees_paid: fee.paidAmount }); // stale value! fee.payments.push({ amount: Number(amount), ... }); await fee.save(); // hook sets paidAmount correctly await User.findByIdAndUpdate(..., { fees_paid: fee.paidAmount }); // now accurate
  • Remove the fee.paidAmount += Number(amount) line entirely
  • Move the User.findByIdAndUpdate sync to after await fee.save() to read the hook-corrected value
  • Add a unit test for the allocatePayments hook to prevent future regression
CRITICAL

Race condition in roll number generation — concurrent creations produce duplicate roll numbers

models/User.js — pre-save hook rollNumber generator
Race conditionConcurrencyE11000
⌄

The roll number generator reads the current count of users with that role, computes a serial, then checks uniqueness in a while loop. Between countDocuments and findOne, two concurrent requests — e.g. admin creating a student while a counsellor converts a lead — get the same count and both generate the same serial. One hits the unique index and throws an unhandled E11000 crash, producing a raw 500 error page.

// Non-atomic — window for collision between count and insert let count = await User.countDocuments({ role: this.role }); rollNum = `VD-${roleCode}-${courseCode}-${year}-${serial}`; // Atomic counter document — no window for collision const counter = await Counter.findByIdAndUpdate( `rollNumber_${roleCode}_${year}`, { $inc: { seq: 1 } }, { new: true, upsert: true } ); this.rollNumber = `VD-${roleCode}-${courseCode}-${year}-${String(counter.seq).padStart(3, '0')}`;
  • Create a Counter model and use an atomic findByIdAndUpdate + $inc to generate serials
  • Wrap user creation in a try/catch that returns a friendly, user-facing error on E11000
  • Keep the existing unique: true, sparse: true index on rollNumber as a safety net
